# Rapid-MLX 이미지 분석 무한반복 근본 수정 계획

## Summary

LLeM은 첨부 이미지를 정상 전달했고 Rapid-MLX도 이미지를 읽어 본문을 생성했다. 실제 문제는 생성 말미에서 `stone stone stone...` 토큰 반복이 발생했고, LLeM이 watchdog 중단 후 같은 `editMessage` 요청을 자동 재시도하면서 동일 루프가 반복된 것이다.

이 수정은 세 축으로 진행한다.

- Rapid-MLX 이미지/OCR 요청은 더 보수적인 샘플링으로 보내 반복 가능성을 낮춘다.
- watchdog이 감지한 모델 출력 반복은 자동 재시도하지 않고, 깨끗한 부분 결과를 보존한 뒤 종료한다.
- 내부 재시도 힌트(`[SYSTEM HINT]...`)가 사용자 메시지, 히스토리, 제목에 노출되지 않게 제거한다.

## Key Changes

- `RepetitionWatchdog` 결과를 구조화한다.
  - `kind`: `token-spam`, `block-repeat`, `action-tag-loop` 등 반복 유형
  - `repeatedToken`: 예: `" stone"`
  - `retryable`: 모델 출력 반복은 `false`
  - `cleanText`: 반복 토큰 꼬리를 제거한 부분 결과
- `StreamOutcome`에 반복 메타데이터를 추가한다.
  - `repeatedKind?: string`
  - `repeatedToken?: string`
  - `retryable?: boolean`
  - `cleanText?: string`
- `chatPipeline`은 반복 감지 시 기존처럼 빈 메시지를 만들지 않는다.
  - `cleanText`가 의미 있는 길이면 부분 답변으로 저장
  - 반복 토큰 꼬리는 히스토리와 UI에서 제거
  - 사용자에게 “반복 토큰이 감지되어 자동 재시도 없이 중단했다”는 짧은 안내 표시
- `sidebarChatProvider`의 `_handleExecutionResult`에서 모델 출력 반복은 자동 retry 금지.
  - 현재처럼 같은 요청을 3초/10초/30초 뒤 재큐잉하지 않음
  - `[SYSTEM HINT]`를 prompt에 덧붙이는 로직 제거
  - 수동 재시도는 사용자가 직접 다시 보낼 때만 가능
- `RequestRetryGuard`는 남겨두되 역할을 “즉시 중복 실행 차단”으로 축소.
  - fingerprint 계산 시 `[SYSTEM HINT]...` 같은 내부 문구는 무조건 제거
  - 같은 이미지/같은 prompt/같은 model의 반복 중단 직후 즉시 재실행은 TTL 동안 차단
- Rapid-MLX 이미지 요청에는 전용 safe sampling profile 적용.
  - 조건: `endpoint.engineKind === "rapid-mlx"` 그리고 첨부 이미지 1개 이상
  - 기본값: `temperature = 0.2`, `top_p = 0.85`, `top_k = 20`, `repeatPenalty = 1.12`, `max_tokens = 2048`
  - 사용자가 명시한 값이 있더라도 이미지/OCR 요청에서는 safe profile 우선 적용
  - `aiClient`의 OpenAI-compatible/Rapid-MLX body에도 `repetition_penalty`와 `top_k`를 전달
- 로그를 강화한다.
  - request_start에 `samplingProfile`, `hasImages`, `repeatPenalty`, `maxTokens` 기록
  - watchdog 중단 시 `repeatedKind`, `repeatedToken`, `cleanChars`, `retryable=false` 기록
  - 자동 재시도 금지 시 `[QUEUE] Repetition stop is non-retryable; not scheduling retry` 기록
- README `🚀 What's New`에 이번 수정 내용을 상세히 추가한다.

## Test Plan

- Unit tests
  - `RepetitionWatchdog`가 `stone stone...` 토큰 반복을 `token-spam`으로 감지하고 반복 꼬리를 제거하는지 확인
  - 반복 감지 결과가 `retryable=false`로 전달되는지 확인
  - `RequestRetryGuard` fingerprint가 `[SYSTEM HINT]` 유무와 관계없이 같은 요청으로 계산되는지 확인
  - Rapid-MLX 이미지 요청 body에 `temperature`, `top_p`, `top_k`, `repetition_penalty`, `max_tokens`가 safe profile로 들어가는지 확인
- Queue behavior tests
  - `result.repeated=true` 및 `retryable=false`일 때 pending retry가 생성되지 않아야 함
  - UI/히스토리/세션 제목에 `[SYSTEM HINT]`가 남지 않아야 함
  - clean partial response가 있으면 AI 메시지로 저장되어야 함
- Manual acceptance
  - Rapid-MLX `gemma-4-26b` + 첨부 이미지 분석 요청 실행
  - 이미지 내용은 인식되어야 함
  - 모델이 반복 토큰에 빠지면 한 번만 중단되고 자동 재시도되지 않아야 함
  - Output log에 `Scheduled retry`가 나오지 않아야 함
  - 채팅창에 `[SYSTEM HINT]`가 보이지 않아야 함
  - 저장된 히스토리 제목에도 `[SYSTEM HINT]`가 없어야 함
- Verification commands
  - `npm test`
  - 필요 시 반복 관련 테스트만 먼저 실행 후 전체 테스트 실행

## Assumptions

- 이미지 인식 자체는 현재 정상 동작하는 상태로 본다.
- 모델 출력 반복은 앱이 자동으로 “고쳐서 재시도”할 수 있는 문제가 아니라, 사용자에게 부분 결과를 보존해 보여주고 멈춰야 하는 generation failure로 취급한다.
- Rapid-MLX 이미지/OCR 요청은 창의적 생성보다 안정성이 우선이므로 safe sampling profile을 강제 적용한다.
- 자동 retry가 필요한 네트워크/일시 오류 처리는 이번 범위가 아니며, 이번 수정은 watchdog 반복 루프에만 적용한다.
